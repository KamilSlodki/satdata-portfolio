## Sistema operativo común

Los 10 showcases comparten un mismo sistema operativo de ingeniería, replicado y evolucionado cross-proyecto a lo largo de 13+ proyectos del ecosistema. La documentación es **activo de primera clase, no afterthought**.

### 5 protocolos de ingeniería

| Protocolo | Qué cubre |
|---|---|
| **Automation Engineering Protocol** | Stage-gate horizontal de cambio: change-spec → guard layer → release train → rollback playbook. |
| **Prompt Engineering Protocol** | Diseño modular de prompts con estructura fija: required/optional/exclusion keywords + patterns regex + confidence scoring + anti-echo + whitelist validator + schema JSON validation. |
| **Post-Development Verification Protocol** | 4 niveles de verificación tras cada cambio: estática + integración + canarios E2E con datos sintéticos + observabilidad diaria. |
| **Post-Development Gates** | 94 gates pre-merge / pre-deploy / post-deploy con checklist bloqueante. |
| **Frozen Zones & Regression Prevention** | Distinción explícita CONFIG (estable) vs HEALTH (móvil), auditoría holística periódica, mapa de zonas frozen para prevenir regresión cross-pieza. |

### Sistema documental replicado

5 capas documentales en cada proyecto del ecosistema, proyectadas al showcase público con restricciones de anonimización:

| Capa | En el proyecto privado | En el showcase público |
|---|---|---|
| **arc42** | 12 secciones (objetivos · restricciones · contexto · estrategia · bloques · runtime · despliegue · cross-cutting · decisiones · calidad · riesgos · glosario) | Secciones 1-7 condensadas en `docs/overview.md` + `docs/architecture.md` |
| **C4** | Context + Container + Component + Code | Context + Container vía Mermaid (render nativo en GitHub) |
| **ADR** | Plantilla fija: contexto · decisión · alternativas · consecuencias · plan rollout · plan rollback | 3-5 ADRs representativas citadas en el README |
| **Runbooks** | Procedimientos operativos: disparadores · precondiciones · severidad · owner · pasos | Mencionados con nombre genérico en `docs/documentation-protocol.md` |
| **Postmortems** | Cronología · causa raíz (5 whys) · acción correctiva → ADR · métrica de prevención | Patrones proyectados en sección "Retos técnicos resueltos" del README |

Cada showcase público lleva su propio `docs/documentation-protocol.md` que enlaza a los artefactos reales que sí están publicados.

### Disciplina cross-proyecto

- **Engineering-playbook propio** con 60+ archivos doctrinales cross-proyecto, evolucionados iterativamente desde incidentes reales (no copy-paste de buenas prácticas genéricas).
- **`verify-findings` adversarial**: cualquier auditoría / hallazgo pasa por un segundo agente independiente antes de cerrar. Ratio de ruido objetivo < 15 %.
- **Sin emojis decorativos · tono operativo y directo · español como idioma de trabajo · "lo que no aporta evidencia, fuera"**.
- **Convenciones de naming explícitas**: `-showcase` para producto propio operado · `-case-study` para entrega cerrada a cliente bajo NDA.
